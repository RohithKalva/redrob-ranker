# Redrob Hackathon — Intelligent Candidate Discovery & Ranking

**Team:** Rohith  
**Challenge:** INDIA.RUNS × Redrob Data & AI Challenge  
**Task:** Rank the top 100 candidates from a 100K pool for a Senior AI Engineer JD

---

## Architecture Overview

```
candidates.jsonl
      │
      ▼
feature_extractor.py          ← run ONCE offline (~10 min)
      │
      ├─ features.pkl          (8 features × 100K rows, ~120 MB)
      ├─ bm25_index.pkl        (BM25 text index)
      └─ honeypot_ids.pkl      (flagged impossible profiles)
      │
      ▼
train_ltr.py                  ← run ONCE offline (~30 sec)
      │
      └─ ranker.pkl            (LightGBM LambdaRank, ~2 MB)
      │
      ▼
rank.py --candidates ./candidates.jsonl --out ./submission.csv
      │
      └─ submission.csv        (100 rows, ranks 1-100)
```

### Signal pipeline

| # | Feature | Weight | What it captures |
|---|---------|--------|-----------------|
| 1 | `skill_match` | 0.30 | Hard/preferred JD skills × proficiency × endorsements |
| 2 | `career_narrative` | 0.22 | Retrieval/ranking vocabulary in role descriptions |
| 3 | `product_company_ratio` | 0.15 | % career at non-consulting product companies |
| 4 | `years_in_retrieval` | 0.10 | Months in retrieval/search/ranking roles |
| 5 | `availability` | 0.10 | Recency + response rate + open_to_work + notice period |
| 6 | `location_fit` | 0.05 | Proximity to Pune/Noida/Hyderabad/Mumbai/Delhi NCR |
| 7 | `experience_fit` | 0.05 | 5–9 YOE ideal band; penalizes title-chaser tenure |
| 8 | `github_verification` | 0.03 | GitHub activity + profile verification |

**Availability multiplier** (0.15–1.0) applied to LightGBM score post-ranking:
- Inactive >180 days → 0.4×
- Response rate <10% → 0.5×
- Notice period >120 days → 0.7×

**Honeypot detection** excludes profiles with impossible signals before top-100 selection.

## Quickstart

### 1. Install dependencies

```bash
pip install -r requirements.txt
```

### 2. Pre-compute features (run once)

```bash
python feature_extractor.py --candidates ./candidates.jsonl --out_dir ./artifacts
```

This produces `artifacts/features.pkl`, `artifacts/bm25_index.pkl`, and `artifacts/honeypot_ids.pkl`.

### 3. Train the LightGBM ranker (run once)

```bash
python train_ltr.py --artifacts ./artifacts --out ./artifacts/ranker.pkl
```

### 4. Rank candidates

```bash
python rank_standalone.py --candidates ./candidates.jsonl --out ./submission.csv
```

**This is the single reproduction command for Stage 3 review.**  
Runtime: ~30–60 seconds on a 16 GB CPU machine with no network access.

### 5. Validate the submission

```bash
python validate_submission.py submission.csv candidates.jsonl
```

### 6. Check for honeypots

```bash
python honeypot_checker.py --submission ./submission.csv --candidates ./candidates.jsonl
```

## Design Decisions

**Why BM25 shortlist + LightGBM re-rank instead of pure embedding search?**  
The 5-minute CPU constraint makes per-candidate embedding inference impractical at 100K scale.
BM25 efficiently narrows to a 2K shortlist; LightGBM then re-ranks using richer features
(behavioral signals, career narrative, company type) that pure BM25 can't capture.

**Why not call an LLM during ranking?**  
Explicitly prohibited by the compute constraints. Our reasoning strings are generated
deterministically from structured profile fields, so they're grounded in facts and
never hallucinate skills the candidate doesn't have.

**Why the availability multiplier?**  
The JD explicitly states: *"A perfect-on-paper candidate who hasn't logged in for 6 months
and has a 5% response rate is not actually available."* This multiplier implements that
directly rather than hoping the model learns it from features.

**How are honeypots avoided?**  
Explicit rule-based checks on impossible profile signals (skill count anomalies,
tenure vs career history mismatch, expert skill inflation). These are excluded before
the final top-100 is selected, not just penalized in scoring.
